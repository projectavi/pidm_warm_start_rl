# Agents Guide

## Purpose

This repository contains the code used to reproduce the 2D navigation experiments from the paper "When does predictive inverse dynamics outperform behavior cloning?". The main package is `pidm_imitation`, and most work falls into one of three areas:

- training imitation-learning models in `pidm_imitation/agents`
- running or extending the toy navigation environment in `pidm_imitation/environment/toy_env`
- evaluating trained checkpoints through `pidm_imitation/toy_evaluate_model.py` and `pidm_imitation/evaluation`

## Tech Stack

- Python 3.10+
- PyTorch 2.3.1
- Lightning 2.5.2
- Gymnasium / Pygame for the toy environment
- Weights & Biases for optional experiment logging

Install from the repo root in a fresh Python 3.10 environment:

```bash
pip install -e .
```

## Repository Map

### Primary Source Directories

- `pidm_imitation/agents/`: model definitions, training loop, dataset/datamodule code, inference agents
- `pidm_imitation/environment/toy_env/`: toy environment, data collection, trajectory serialization, configs
- `pidm_imitation/evaluation/`: evaluation CLI helpers, rollout logic, toy-agent wrappers
- `pidm_imitation/configs/`: config parsing helpers and config object definitions
- `pidm_imitation/utils/`: logging, IO, controller, trajectory, and config utilities

### Config and Template Directories

- `configs/supervised_learning/`: hand-authored example training configs such as `bc_example.yaml` and `pidm_example.yaml`
- `configs/toy_env/`: hand-authored toy environment configs
- `oss_configs/templates/`: source templates for bulk-generated experiment configs
- `toy_configs/`: generated config files; prefer regenerating these from `oss_configs/templates/` via `oss_configs/generate_configs_human.py` instead of hand-editing hundreds of files

### Data and Generated Output

Treat these as data or generated artifacts unless the task explicitly targets them:

- `datasets/`: released training/evaluation datasets
- `checkpoints/`: model checkpoints
- `logs/` and `logs/wandb/`: training logs
- `my_experiment/`: recorded toy-environment trajectories
- `rollout_*` directories: evaluation outputs

Do not mass-edit or delete generated artifacts unless the user asks for that specifically.

## Common Entry Points

### Record Toy-Environment Data

```bash
python -m pidm_imitation.environment.toy_env.play \
  --config configs/toy_env/toy_env_four_room.yaml \
  --agent random \
  --episodes 1
```

To record trajectories, add `--record --experiment <name>`. Human collection requires a supported Xbox controller.

### Train a Model

```bash
python -m pidm_imitation.agents.supervised_learning.train \
  --config configs/supervised_learning/bc_example.yaml \
  --new
```

Notes:

- training configs are config-driven; prefer changing YAML and config classes over hard-coding behavior
- example configs default to `accelerator: gpu`; switch to CPU in the YAML for local smoke runs on machines without CUDA
- checkpoints default to `checkpoints/`, and logs go to `logs/`

### Evaluate a Checkpoint

```bash
python -m pidm_imitation.toy_evaluate_model \
  --toy_config configs/toy_env/toy_env_four_room.yaml \
  --config configs/supervised_learning/pidm_example.yaml \
  --checkpoint checkpoints/last.ckpt \
  --agent toy_idm \
  --episodes 1 \
  --save_results
```

Evaluation creates `rollout_<checkpoint>_<n>/` folders under `--output_dir` and can optionally save trajectories, video, and `results.json`.

## Working Rules for Agents

### Favor the Existing Architecture

- Use the existing factory/config pattern instead of introducing one-off constructor logic.
- New model behavior usually belongs under `pidm_imitation/agents/supervised_learning/` plus config wiring.
- New toy-environment behavior usually belongs under `pidm_imitation/environment/toy_env/` and its config classes.
- Reuse `pidm_imitation.utils.Logger` and existing utilities before adding new helpers.

### Keep Configs as the Control Surface

- Training and evaluation behavior is mostly driven by YAML plus config wrappers such as `OfflinePLConfigFile` and toy-env config classes.
- If a feature changes runtime behavior, prefer exposing it through config parsing rather than embedding repo-local paths or constants.
- Preserve the current file naming conventions for checkpoints and trajectory assets unless you also update the evaluation utilities that discover them.

### Maintain Handoff Memory

- Keep `IMPLEMENTATION_PROGRESS_HANDOFF.md` updated during implementation work.
- Update it when starting a task, after meaningful progress, and when pausing or finishing.
- Use it to record active objectives, touched files, blockers, validation status, and next steps.

### Be Careful With Generated Configs

- `toy_configs/` is bulk-generated by `oss_configs/generate_configs_human.py`.
- If the change should apply broadly across those configs, edit the template(s) in `oss_configs/templates/` and regenerate.
- Only hand-edit a generated config when the task is explicitly about that one file.

## Validation

There is no dedicated test suite checked into this repository. Use targeted validation instead:

- run `python -m compileall pidm_imitation` for a quick syntax check after Python changes
- run the smallest possible CLI smoke test for the area you changed
- for training changes, reduce `max_steps` and use an existing example config
- for evaluation changes, run a single episode with `--episodes 1`

## Practical Cautions

- The repo may contain large datasets and previously generated experiment outputs; avoid touching them unless necessary.
- `wandb` is enabled by config in some examples. Use offline mode or disable it when you only need a local smoke test.
- README examples are mostly accurate, but prefer the current code paths and flags in `train.py`, `play.py`, and `toy_evaluate_model.py` when they disagree with older docs.
